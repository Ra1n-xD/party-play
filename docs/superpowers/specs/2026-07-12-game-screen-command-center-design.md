# Редизайн GameScreen по макетам «Бункер»

Дата: 2026-07-12

Статус: утверждено пользователем

Область изменений: только основной игровой экран `GameScreen` и его адаптивные стили.

## Цель

Привести основной игровой экран к предоставленным desktop- и mobile-макетам: тёмный зелёно-графитовый интерфейс командного центра, компактная сводка состояния, сопоставимые карточки всех участников и фиксированная панель доступных действий. Игровая логика, серверное состояние, сокет-события, маршрутизация фаз и пользовательские сценарии не меняются.

## Границы задачи

Изменяются:

- JSX-разметка `GameScreen`;
- локальные вычисления, необходимые только для отображения существующих данных;
- стили, ограниченные корневым классом нового игрового экрана;
- адаптивность desktop, tablet и mobile;
- представление уже существующих модальных окон и панели управления хоста внутри `GameScreen`.

Не изменяются:

- `VoteScreen`, `LobbyScreen`, `ResultsScreen`, `HomeScreen`;
- `GameContext`, Socket.IO-контракты и сервер;
- правила раскрытия характеристик, голосования, паузы и завершения игры;
- глобальные оверлеи раскрытия карт и объявления фаз;
- набор существующих действий и проверок доступа.

## Визуальное направление

- Основной фон почти чёрный с зелёным оттенком.
- Поверхности карточек — графитово-зелёные, с тонкими полупрозрачными границами.
- Главный акцент — тёплый золотой; опасные действия — приглушённый красный.
- Каждый игрок получает устойчивый локальный цветовой акцент из ограниченной палитры.
- Типы характеристик продолжают различаться цветом и существующей иконографикой `CardImage`.
- Типографика компактная, плотная и хорошо читаемая на мобильном экране.
- Декоративные фоновые частицы скрываются только в пределах игрового экрана визуально за непрозрачным фоном; сам глобальный компонент не меняется.

## Структура экрана

### Верхняя панель

В шапке отображаются существующие данные и действия:

- бренд «Бункер»;
- код комнаты;
- состояние подключения;
- кнопка управления хоста, открывающая существующую админ-панель;
- выход из комнаты через существующий `leaveRoom`.

Элементы из макета, для которых нет обработчика, не создаются.

### Сводка состояния

Единый блок содержит:

- название и описание катастрофы;
- таймер текущей серверной фазы;
- вместимость бункера и число живых игроков;
- номер текущего раунда;
- прогресс раундов;
- фазовую инструкцию;
- раскрываемые существующие сведения о картах бункера и угрозе.

Тексты таймера остаются фазово корректными: «До голосования» используется только во время обсуждения; в остальных фазах отображается нейтральная подпись текущего этапа.

### Участники

Под сводкой располагаются заголовок, агрегированное количество открытых характеристик и легенда статусов. На мобильном экране добавляется горизонтальная лента участников как визуальный обзор списка без новой игровой функции.

### Карточка игрока

На desktop карточки располагаются в четыре колонки, на tablet — в две, на mobile — в одну. Каждая карточка содержит:

- инициалы, имя, номер, признаки `BOT` и «Вы»;
- подключение, изгнание, право голоса последнего изгнанного;
- состояние текущего хода;
- семь фиксированных строк: шесть типов характеристик и особое условие;
- раскрытые значения или явно заблокированный placeholder;
- прогресс раскрытия и существующий статус голосования, если он доступен в публичном состоянии.

Для текущего игрока приватные данные берутся только из `myCharacter`. Они видны владельцу независимо от публичного раскрытия, а нераскрытые строки получают визуальный признак приватности. Для остальных игроков используются только `revealedAttributes` и раскрытая `actionCard`; недоступные значения не попадают в DOM.

Клик по карточке сохраняет существующее подробное модальное представление.

### Нижняя панель

Фиксированная панель содержит только реально доступные существующие действия:

- раскрытие характеристики;
- раскрытие особого условия;
- фазовое действие хоста «Пропустить обсуждение»;
- доступ к управлению хоста, если это требуется компоновкой.

Кнопки «Пауза» и «Голосование» из макета не добавляются как новые действия. Пауза продолжает управляться существующим открытием/закрытием админ-панели, а голосование остаётся отдельной серверной фазой и `VoteScreen`.

## Адаптивность

### Desktop, от 1200 px

- четыре карточки игроков в строке;
- сводка состояния строится в одну широкую панель;
- нижняя панель показывает контекст хода слева и действия справа;
- высота карточек выравнивается по содержимому без обрезания текста.

### Tablet, 769–1199 px

- две карточки игроков в строке;
- метрики сводки переносятся на следующую строку;
- элементы управления сохраняют сенсорную область не менее 44 px.

### Mobile, до 768 px

- компактная шапка без второстепенных подписей;
- сводка разбивается на сценарий, две метрики и прогресс;
- номер раунда скрывается из отдельной метрики, если не помещается, но остаётся в прогрессе;
- горизонтальная лента участников не расширяет страницу;
- карточки идут одной колонкой;
- фиксированная нижняя панель учитывает `safe-area-inset-bottom`;
- контент имеет достаточный нижний отступ и не перекрывается панелью;
- отсутствует горизонтальная прокрутка на ширине 320 px.

## Существующие состояния и ограничения

- Для зрителя приватные данные и игровые действия отсутствуют.
- При временном отсутствии `myCharacter` показывается безопасное состояние загрузки вместо пустого экрана.
- Изгнанные и отключённые игроки остаются различимыми текстом и стилем.
- Админские операции, подтверждения, раскрытие карт и завершение игры сохраняют существующие обработчики.
- Все данные остаются server-authoritative; новые локальные вычисления не изменяют состояние игры.

## Проверка

Автоматически:

1. `npm -w client run build`.
2. `npm -w server run build` как контроль отсутствия затронутых общих типов.
3. `npm run format:check`.

Визуально:

- 1440×900 — четыре колонки и полная нижняя панель;
- 1024×768 — две колонки без переполнения;
- 390×844 — мобильный viewport и фиксированное действие;
- 320 px — отсутствие горизонтального скролла;
- длинная мобильная страница — последовательный список всех игроков.

Функционально проверяются существующие сценарии:

- раскрытие обязательной и выбранной характеристики;
- раскрытие особого условия с подтверждением;
- открытие подробной карточки игрока;
- открытие и закрытие управления хоста с паузой/возобновлением;
- пропуск обсуждения хостом;
- завершение игры;
- выход из комнаты;
- отображение зрителя, изгнанного и отключённого игрока.

## Критерии приёмки

1. `GameScreen` визуально максимально близок к предоставленным макетам.
2. Другие экраны и глобальные игровые оверлеи не меняются.
3. Все существующие ограничения ролей и обработчики действий сохранены.
4. Приватные характеристики никогда не отображаются другим игрокам.
5. На desktop карточки идут в четыре колонки, на mobile — в одну.
6. Нижняя панель не перекрывает последнюю карточку и модальные действия.
7. На ширине 320 px нет горизонтальной прокрутки.
8. Клиентская и серверная сборки проходят успешно.
